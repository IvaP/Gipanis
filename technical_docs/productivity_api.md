- HTTP request
    - загальний формат URI: `/machines/{machine_name}/productivity?start={start_timestamp}&end={end_timestamp}&bucket={bucket_value}`
        - `machine_name` — назва верстата. Наприклад: `Eva1`
        - `start_timestamp`, `end_timestamp` — дата_час початку та кінця вибірки відповідно
            - __значеннями є дата_час у форматі ISO 8601/UTC+0 з точністю до секунд.__ Наприклад: `2026-05-03T20:00:00Z`.<br>При конвертації локального дата_часу в даний формат потрібно брати до уваги поточну часову зону
        - `bucket_value` — величина проміжків часу на які буде дробитись загальна вибірка; продуктивність обраховується в контексті кожного з цих bucket'ів
            - допустимі значення `bucket_value`:
                - `1m`
                - `15m`
                - `30m`
                - `1h`
                - `2h`
                - `6h`
                - `12h`
                - `1d`
        - приклад URI: `/machines/EVA1/productivity?start=2026-05-02T10:00:00Z&end=2026-05-02T20:00:00Z&bucket=1h`
    - токен
        - токен зберігається в HTTP header'і під назвою `Authorization`
        - значення header'а `Authorization`: `Bearer yT4oJneVwBaQMJeO8TiUN1qycBf9AGem`
- HTTP response
    - у відповідь сервер присилає JSON-масив об'єктів наступного вигляду
      ```json
      [
        {
          "end_bucket_timestamp": "2026-04-01T01:00:00Z",
          "ws1_l": 0,
          "ws1_r": 0,
          "ws2_l": 0,
          "ws2_r": 0,
          ...
        },
        {
          "end_bucket_timestamp": "2026-04-01T02:00:00Z",
          "ws1_l": 14,
          "ws1_r": 13,
          "ws2_l": 10,
          "ws2_r": 11,
          ...
        },
        ...
      ]
      ```
        - кожний з об'єктів-словників є відображенням відповідного дато_часового bucket'а
          - `end_bucket_timestamp` — календарний дата_час <u>закінчення</u> bucket'а
            - __задається у форматі ISO 8601/UTC+0 з точністю до секунд.__ При конвертації цього значення у локальний час потрібно брати до уваги поточну часову зону
          - `wsn_l` — продуктивність n-го робочого місця( `ws` скорочено від `work station` ) лівої форми( `l` - `left` )
          - `wsn_r` — продуктивність n-го робочого місця( `ws` скорочено від `work station` ) правої форми( `r` - `right` )
    - __при аналізі даної вибірки необхідно враховувати, що дато_часові проміжки будуються не по відношенню до заданих в URI `start_timestamp` та `end_timestamp`, а по відповідних дато_часових календарних проміжків.__<br>
      Наприклад, якщо `start_timestamp=2026-05-03T20:10:00Z`, `end_timestamp=2026-05-03T22:10:00Z`, `bucket_value=1h`, то фактичний початок першого bucket'а буде `2026-05-03T20:00:00Z`, а кінець останнього — `2026-05-03T23:00:00Z`.
      Як наслідок останній bucket може бути неповним( мається на увазі, що на сам момент вибірки він ще не буде повністю заповнений даними )
    - у "пустих" backet'ів( ті, що не містять у собі жодних даних ) значення `wsn_l`та `wsn_r` будуть дорівнювати `0`
